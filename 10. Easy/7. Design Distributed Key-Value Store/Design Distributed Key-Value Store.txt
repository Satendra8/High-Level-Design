        1. Data Partition

            split data in small partitions and move to multiple Servers

            Two challenges:
                -> Distribute data across multiple servers evenly
                -> minimize data movement when nodes are added or removed

            Consistent Hashing (use virtual server to avoid un even distribution)


        2. Data Replication

            for high Availability and Reliability
            data must be replicated to n server on different locations


        3. Consistency

            data must be same on all nodes at a time

            Tunable consistency

                N: number of replicas
                W: write quorum of size W
                R: read quorum of size R

                W: for write operation to be successful, when W replicas ACKNOWLEDGE

                R: for read operation to be successful, wait for the response of R replica


                W = 1: fast write
                R = 1: fast read

                W or R > 1: consistency high, but latency high

                W+R > N: strong consistency guranteed



                Eg: W = 2, R = 2, N = 3

                    4 > 3  strong consistency guranteed




                R = 1, W = N (fast read)
                R = N, W = 1 (fast write)
                W + R > N (strong consistency)


            depending on Requiremment you can tune W, R and N.

        

            Consistency Models:

                i. Strong Consistency: client never see out dated data
                ii. Weak Consistency: subsequent read operation may not see latest data always
                iii. Eventual Consistency: Given enough time, all updated propogated to replica and they are updated


            
            What we be used in key value store?

                we shall use Eventual Consistency

                    -> High availability
                    -> Low latency
                    -> Horizontal scalability

                Dynamo & Cassendra also adopted Eventual Consistency

        5. Handling Failures

            i. Temporary Failures
            ii. Permanent Failures


            Failute Detection:

                i. All to all Multicasting: each server node send message to each one, if not responded then there is a failure

                ii. Gossip Protocol:

                    -> Every node has a membership list [memberId, Heartbeat Counter, Time]

                    for S0 node MemberShip list

                    memberId        Heartbeat Counter      Time
                       0                  10232          12:00:01
                       1                  10224          12:00:10
                       2                  9000           11:50:02


                    S0 keep updating its heartbeat and send its Membership list to other random nodes (Say S1)
                    S1 compare time and update its Membership list
                    If a Node has not updated and it exceed a Fixed timeout mark dead
                    S1 sends to other nodes its MemberShip list, then they also know that a node is dead

        
            Temporary Failure:

                Strict Quorum: Read/Write blocked

                Sloppy Quorum: Used to improve availability

                    Hash Ring:
                        First W servers to write
                        First R servers to read


                    When server is down, some other server will process request temporarily.
                                                                              
                    Once it is up, changes will be pushed back to achieve consistency. | Hinted Handoff



            Permanent Failure:

                We implement Anti-entropy protocol to keep replicas in sync.

                Anti-entropy:
                    Compare each piece of data on replicas and updating them.
                    We use MERKLE TREE


                    MERKLE TREE:
                        1. create hash for all data
                        2. add up and make tree
                        3. compare tree of each server

                        benifit: fast comparision


                Handling data center Outage

                    -> Power outage
                    -> Network outage
                    -> Natural Disaster

                    We should replicate data across multiple data centers
